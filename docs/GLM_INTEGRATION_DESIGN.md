# GLM Integration Design

## Overview

Add GLM (Generalised Linear Model) support to Haute's modelling pipeline, using RustyStats as the fitting engine and Atelier as the interactive workbench.

## Architecture: Two Independent Tools

```
Atelier (interactive)              Haute (automated)
─────────────────────              ─────────────────
Fit GLM manually           ───►   RustyStats dict API code
  │                                (stored in node config)
  │                                      │
Export dict API code               Execute dict API code
  │                                      │
Log model to MLflow                Log model to MLflow
         │                               │
         └──────────┬────────────────────┘
                    │
              MLflow Registry
                    │
              MODEL_SCORE node
              (loads from MLflow, algorithm-agnostic)
```

- **Atelier** — standalone interactive GLM workbench for actuaries to build GLMs with full statistical analysis (variable selection, banding, diagnostics, residual plots, etc.)
- **Haute** — pipeline engine that can re-train GLMs from a saved spec and score with fitted models
- **MLflow** — universal model registry; the handoff point between both tools

Atelier is NOT embedded in Haute. The actuarial GLM workflow (iterative variable selection, in-depth diagnostics, coefficient inspection) is a fundamentally different interaction pattern from pipeline building and warrants its own dedicated tool.

## Libraries

### RustyStats

Rust-backed GLM engine with Python bindings (`github.com/PricingFrontier/rustystats`).

- **Dict API**: `rustystats.glm_dict(response, terms, data, family, offset).fit()`
- **Families**: gaussian, poisson, binomial, gamma, tweedie, quasipoisson, quasibinomial, negbinomial
- **Features**: splines, monotonicity constraints, regularization, interactions, target encoding
- **Serialization**: native binary format
- **Dependencies**: numpy, polars (maturin-built Rust extension)

### Atelier

Browser-based GLM workbench wrapping RustyStats (`github.com/PricingFrontier/atelier`).

- FastAPI + React + RustyStats (similar architecture to Haute)
- CLI: `atel` / `atelier`
- Status: **incomplete** — needs finishing before GLM integration into Haute

## Atelier Outputs

1. **RustyStats dict API code** — Python code defining the GLM specification (terms, family, link function, constraints, etc.). This is the portable "recipe" for the model.
2. **Fitted model** — logged to MLflow via native RustyStats serialization.

## Haute Integration Design

### Training Path

The GLM training node in Haute stores the RustyStats dict API code (generated by Atelier or written by hand). When the user trains:

1. Haute executes the dict API code against the upstream data
2. RustyStats fits the GLM
3. The fitted model is logged to MLflow

Implementation:

- Add `GLMAlgorithm(BaseAlgorithm)` to `ALGORITHM_REGISTRY` in `_algorithms.py`
- The algorithm's `fit()` method executes the dict API code
- Same `TrainingJob` pipeline stages as CatBoost (prepare → split → train → evaluate → save)

### Scoring Path (MODEL_SCORE node)

Loads the fitted model from MLflow and scores — identical flow regardless of whether the model was fitted in Atelier or Haute.

### MLflow Strategy: Native Flavors, Not pyfunc

**Decision**: keep native MLflow flavors and dispatch on model type at load time.

Reasons to avoid pyfunc:

| Concern | Native Flavor | pyfunc |
|---------|--------------|--------|
| Serialization | Native binary (`.cbm`, RustyStats format) — fast, compact | pickle/cloudpickle — fragile across Python versions |
| Model introspection | Full access (feature importance, SHAP, coefficients) | Opaque `.predict()` only |
| Categorical metadata | Preserved (CatBoost `get_cat_feature_indices()`) | Lost |
| Classification | `predict_proba()` available | `.predict()` only |
| GPU inference | Native support (CatBoost) | Not guaranteed |
| Databricks Serving | Optimised native support | Generic |

**Approach**:

```
mlflow.catboost.log_model()     →  model_type tag: "catboost"
rustystats MLflow flavor        →  model_type tag: "glm"
                                         │
                              _mlflow_io.py dispatches:
                              ├─ "catboost" → _load_catboost_model()
                              └─ "glm"     → _load_rustystats_model()
```

### Current CatBoost-Specific Code to Refactor

`_mlflow_io.py` is currently hardcoded to CatBoost:

- `_find_cbm_artifact()` — only searches for `.cbm` files
- `_load_catboost_model()` — hardcoded to `CatBoostRegressor` / `CatBoostClassifier`
- `_prepare_predict_frame()` — uses `model.get_cat_feature_indices()`
- `_score_eager()` — calls CatBoost-specific `predict_proba()`

These need to become model-type-aware, dispatching based on a `model_type` tag logged to MLflow.
