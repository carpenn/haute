# Price Optimisation Library — Design Specification

## Overview

A standalone Python library (numpy-only core, no haute dependency) that implements insurance price optimisation. Two modes of operation: **online optimisation** (per-risk optimal pricing via Lagrangian dual decomposition) and **ratebook optimisation** (optimal rating factor levels via coordinate descent with Lagrangian inner loop). Designed to run on CPU across millions of risks, with strict memory efficiency requirements.

Haute wraps this library via two node types: **Optimiser** (batch solve) and **Optimiser Apply** (live scoring). The library itself has no knowledge of haute, Polars, or pipelines.

---

## Architecture

```
haute node (Polars in/out, config, UI hooks)
    ↓
solver library (pure numpy, no haute dependency)
    ↓
numpy / scipy
```

The library is independently installable, testable, and usable outside haute. Haute's optimiser nodes are thin wrappers that handle the Polars ↔ numpy boundary and pipeline integration.

## Core Concepts

### Decision Variable

The decision variable is a **multiplier** applied to the technical price, not the price itself. Typical range: 0.8–1.2 (configurable per use case). This means the optimisation grid is uniform across all risks — every risk is evaluated over the same multiplier steps, even though the underlying technical prices differ.

### Inputs

The optimiser receives pre-computed values from upstream models (GLMs, CatBoost, etc.). It does not fit any models itself. The inputs are:

- **Objective values**: a `(N, M)` array where `objective[i, j]` is the objective value (e.g. expected income) for risk `i` at multiplier step `j`. Computed upstream by evaluating the prediction models at each price point.
- **Constraint values**: one `(N, M)` array per constraint, same structure. e.g. `volume[i, j]` = predicted volume (conversion probability) for risk `i` at step `j`.
- **Multiplier grid**: an `(M,)` array of the multiplier steps (e.g. `[0.80, 0.81, ..., 1.20]`).
- **Constraint thresholds**: one scalar per constraint (e.g. "total volume must be ≥ 90% of baseline").

### Curve Fitting

The upstream pipeline evaluates predictions at M discrete multiplier steps. Before optimisation, the library fits **cubic splines** through these points for each risk, for both the objective and each constraint. This produces smooth, differentiable curves.

Key property: because the multiplier grid is uniform across all risks (same knot positions), the spline fitting is a **batched matrix operation** — not a per-risk loop. The spline basis matrix is computed once and applied to all risks simultaneously.

The fitted splines are used:
- During the batch solve (smooth optimisation landscape)
- As artifacts for live scoring (the Apply node stores spline coefficients)

### Outputs

**Online optimisation:**
- Lambda values: one scalar per constraint (Lagrange multipliers / shadow prices)
- Spline coefficients: per-risk spline parameters for objective and constraints
- These two artifacts are all the Apply node needs for live scoring

**Ratebook optimisation:**
- Updated factor tables: adjusted factor values per rating factor level
- These get applied as modified rating tables in the pricing pipeline

**Both modes also produce:**
- Efficient frontier: the Pareto curve of objective vs constraint values, generated by sweeping constraint thresholds
- Impact analysis: before/after price distributions, per-segment breakdown
- Constraint summary: which constraints are binding, how much slack remains

## Online Optimisation Algorithm

### Lagrangian Dual Decomposition

The online optimiser uses Lagrangian dual decomposition. This exploits the separable structure of the problem — once the Lagrange multipliers (lambdas) are fixed, each risk's subproblem is independent.

### Problem Formulation

**Primal problem:**

```
Maximise   Σ_i objective_i(m_i)
Subject to Σ_i constraint_k_i(m_i) ≥ threshold_k   for each constraint k
           m_i ∈ [m_min, m_max]                      for each risk i
```

Where `m_i` is the multiplier for risk `i`, and `objective_i(m)` and `constraint_k_i(m)` are the fitted spline functions.

**Lagrangian relaxation:**

```
L(λ) = Σ_i [ objective_i(m_i) - Σ_k λ_k × (threshold_k - constraint_k_i(m_i)) ]
```

For fixed λ, this decomposes into N independent subproblems:

```
For each risk i: maximise objective_i(m) + Σ_k λ_k × constraint_k_i(m)
```

Each subproblem is a 1D maximisation over the multiplier range — trivially solved by evaluating the lagrangian at the M grid points and taking the argmax (or analytically via the spline derivative).

### Algorithm — Step by Step

```
Input:
  objective:    (N, M) float32 — objective values per risk per step
  constraints:  list of K arrays, each (N, M) float32
  thresholds:   (K,) float64 — constraint thresholds
  step_size:    float — subgradient step size (or use bisection)
  max_iter:     int — max outer iterations (default 50)
  chunk_size:   int — number of risks per memory chunk (default 500_000)

Initialise:
  λ = zeros(K)  # Lagrange multipliers, one per constraint

Outer loop (repeat until convergence or max_iter):

  For each chunk of risks (size chunk_size):

    Step 1 — Compute lagrangian matrix:
      L[i, j] = objective[i, j] + Σ_k λ_k × constraint_k[i, j]
      Shape: (chunk_size, M). Computed in-place to avoid allocation.

    Step 2 — Find optimal step per risk:
      best[i] = argmax_j L[i, j]
      Shape: (chunk_size,). One numpy argmax call.

    Step 3 — Accumulate portfolio totals:
      total_objective += Σ_i objective[i, best[i]]
      total_constraint_k += Σ_i constraint_k[i, best[i]]
      Uses advanced indexing: objective[arange(chunk), best]

  Step 4 — Update lambdas (after all chunks):
    For each constraint k:
      λ_k = max(0, λ_k + step_size × (threshold_k - total_constraint_k))

    Lambdas increase when constraints are violated (pushing the solution
    toward feasibility) and decrease when constraints have slack.

  Check convergence:
    If all constraints satisfied and lambda changes < tolerance → stop.

Output:
  λ values (K,)
  optimal_step per risk (N,) — index into multiplier grid
  spline coefficients per risk (for live scoring)
```

### Lambda Update Strategy

Two options:

**Subgradient method:**
- Simple: `λ_k += step_size × (threshold_k - total_constraint_k)`
- Step size needs tuning (diminishing step size, e.g. `α / sqrt(t)`)
- Convergence can be slow or oscillatory

**Bisection on the dual:**
- For each constraint, binary search for the lambda that makes the constraint bind exactly
- More iterations per update but much tighter convergence
- Each bisection step requires a full portfolio sweep

Recommendation: start with subgradient (simpler), add bisection as an option if convergence is too slow.

### Memory Management

Target: ~200 MB peak regardless of portfolio size.

- **float32 throughout**: sufficient precision for insurance pricing (prices to nearest penny). Halves memory vs float64.
- **Chunked processing**: process risks in chunks of ~500K. The lagrangian matrix is only allocated at chunk size, not full portfolio size.
- **In-place computation**: the lagrangian is computed in-place over a pre-allocated buffer, not by creating new arrays each iteration.
- **Streaming aggregation**: portfolio totals (Step 3) are accumulated across chunks, not stored per-risk.

Memory breakdown for 5M risks, 50 steps, 3 constraints, chunk size 500K:

| Array | Shape | Size (float32) |
|---|---|---|
| objective (full) | (5M, 50) | 1 GB |
| constraints (full) | 3 × (5M, 50) | 3 GB |
| lagrangian buffer | (500K, 50) | 100 MB |
| best indices (full) | (5M,) | 20 MB |
| **Total** | | **~4.1 GB** |

If even 4 GB is too much, the input arrays themselves can be memory-mapped or chunked from disk (parquet → numpy chunks). The solver only needs one chunk at a time.

### Performance Estimate

Per outer iteration:
- One elementwise broadcast per constraint over `(chunk, M)`: ~ms per chunk
- One argmax over `(chunk, M)`: ~ms per chunk
- 10 chunks × overhead: ~50ms per iteration

For 30 outer iterations: **~1.5 seconds** total solve time (5M risks, 50 steps, CPU).

This is conservative — numpy vectorised operations on float32 are extremely fast. Actual time may be sub-second.

## Live Scoring (Online Optimiser Apply)

At quote time, the Apply node receives:
- The risk's features (to evaluate the spline)
- The stored lambda values
- The stored spline coefficients for this risk (or a segment-level approximation)

The calculation per quote:

```
For each multiplier step j:
  L(j) = objective_spline(j) + Σ_k λ_k × constraint_k_spline(j)
optimal_step = argmax L
optimal_multiplier = grid[optimal_step]
optimised_price = technical_price × optimal_multiplier
```

This is a single 1D evaluation — microseconds per quote. No portfolio context needed; the lambdas encode the portfolio constraints.

**Question for live scoring:** the spline coefficients are per-risk, fitted on the batch. For a brand new quote not in the batch, you need to evaluate the upstream models at M price points and fit the splines on the fly, OR use segment-level average splines. The former is more accurate but requires M model evaluations per quote. The latter is faster but coarser.

## Ratebook Optimisation Algorithm

### Coordinate Descent with Lagrangian Inner Loop

The ratebook optimiser adjusts rating factor levels (not individual risk prices). The factor structure means the price for risk `i` is:

```
price_i = base × factor1[level1_i] × factor2[level2_i] × ... × factorF[levelF_i]
```

The decision variables are the factor values at each level of each factor. e.g. 10 factors × 20 levels = 200 variables.

The combinatorial explosion (20^10 possible factor combinations) makes it infeasible to enumerate all possibilities. Instead, use **coordinate descent**: optimise one factor at a time, holding all others fixed, and cycle through factors until convergence.

### Algorithm — Step by Step

```
Input:
  portfolio:     (N,) risk data with factor level indices
  factors:       list of F factors, each with L_f levels
  objective_fn:  function(prices) → portfolio objective value
  constraint_fns: list of K functions, each: function(prices) → constraint value
  thresholds:    (K,) constraint thresholds

Initialise:
  factor_values[f] = current factor values for each factor f

Outer loop (cycle through factors until convergence):

  For each factor f:

    Hold all other factors fixed. Compute the "residual price" for each risk:
      residual_i = base × Π_{g ≠ f} factor_g[level_g_i]

    Now the price for risk i depends only on factor f:
      price_i = residual_i × factor_f[level_f_i]

    Group risks by their level of factor f.
    For each level l of factor f, all risks in that group share the same
    factor value. The decision is: what value should factor_f[l] take?

    This is a small Lagrangian problem:
      - Decision variables: L_f factor values (e.g. 20 values)
      - Each candidate value affects all risks at that level
      - Use the same Lagrangian machinery: evaluate objective/constraints
        at candidate factor values, find the optimal per-level

    Update factor_f with the optimised values.

  Check convergence:
    If factor changes < tolerance across a full cycle → stop.
```

### Per-Factor Subproblem

For factor `f` with `L` levels, and `N_l` risks at each level:

- Generate candidate values for each level (e.g. current value × multiplier grid)
- For each candidate, compute the resulting prices for all risks at that level
- Evaluate objective and constraints
- Use the Lagrangian to find optimal values subject to portfolio constraints

This reuses the same Lagrangian dual decomposition core — the "risks" are now the factor levels, and the "steps" are the candidate values. The inner solve is tiny (L levels × M candidate steps) but the function evaluation sweeps the full portfolio.

### Performance Estimate

- Per factor: one portfolio sweep to compute residuals, one Lagrangian solve (small), one portfolio evaluation
- Per cycle: F factors × O(N) per factor
- Convergence: typically 5–10 cycles
- For 10 factors, 5M risks: ~50 portfolio sweeps × ~1 second each = **~50 seconds**

## Efficient Frontier Generation

The efficient frontier is the Pareto curve showing the tradeoff between objective and constraints. Generated by solving the optimisation at multiple constraint threshold levels.

### Method

```
For threshold in linspace(min_feasible, max_feasible, n_points):
  solve optimisation with this threshold
  record (total_objective, total_constraint_values)

Plot the resulting curve.
```

Each point on the frontier is a full solve. With n_points = 20 and a 1.5-second solve time: **~30 seconds** for the full frontier.

The frontier is computed once (batch) and displayed in the UI. The actuary picks a point on the frontier, which determines the constraint thresholds, which determines the lambdas, which get deployed for live scoring.

### Node Types

**Optimiser Node** (batch only):

```python
@pipeline.node(
    optimiser=True,
    mode="online",  # or "ratebook"
    multiplier_range=(0.8, 1.2),
    multiplier_steps=50,
    objective="expected_income",
    constraints={"volume": {"min": 0.9}},
)
def optimise_prices(df: pl.LazyFrame) -> pl.LazyFrame:
    ...
```

Inputs:
- A DataFrame with columns for objective and constraint values at each multiplier step (produced by upstream prediction nodes evaluated at each step)
- Configuration: multiplier range, steps, objective column, constraint definitions

Outputs:
- Lambda values saved as artifact
- Spline coefficients saved as artifact
- DataFrame with optimised multiplier and price per risk
- Frontier data for UI

**Optimiser Apply Node** (batch + live):

```python
@pipeline.node(
    optimiser_apply=True,
    optimiser_artifact="artifacts/optimiser_v1",
)
def apply_optimised_price(df: pl.LazyFrame) -> pl.LazyFrame:
    ...
```

Inputs:
- A DataFrame with the risk features needed to evaluate splines
- Stored artifacts: lambdas + spline coefficients

Outputs:
- DataFrame with `optimised_multiplier` and `optimised_price` columns

### Batch vs Live Behaviour

| | Batch | Live |
|---|---|---|
| **Optimiser** | Full solve: fit splines, run Lagrangian, extract lambdas, generate frontier | Not used — batch only |
| **Optimiser Apply** | Apply stored lambdas + splines to full portfolio | Apply to single quote (microseconds) |

### Upstream Pipeline Pattern

To feed the optimiser, the upstream pipeline needs to evaluate predictions at M price points per risk. This means the prediction nodes run M times per risk (once per multiplier step). The pattern:

1. Expand the portfolio: each risk × M multiplier steps → `N × M` rows
2. Compute `candidate_price = technical_price × multiplier`
3. Run prediction models (conversion, income, etc.) at each candidate price
4. Pivot back to `(N, M)` shape for the optimiser

This expansion happens before the optimiser node. It could be a built-in capability of the optimiser node (it knows the multiplier grid and can expand internally), or an explicit upstream transform.

## UI Design

### Panel Tabs

**Frontier Tab** (shared by both modes):
- Interactive Pareto curve: drag a point along the frontier
- X-axis: constraint value (e.g. volume retention %)
- Y-axis: objective value (e.g. total expected income)
- Selecting a point updates the constraint thresholds, triggers a re-solve, and updates the other tabs
- Multiple constraints: show 2D frontier for the primary tradeoff, with secondary constraints as binding/non-binding indicators

**Impact Tab** (shared by both modes):
- Before/after price distribution (histogram overlay)
- Per-segment breakdown: which segments see the largest price changes
- Summary statistics: mean change, % of risks with increase/decrease, max change

**Curve Tab** (online mode):
- Per-risk spline visualisation: select a risk, see its objective/constraint curves as a function of multiplier
- Overlay the optimal point (where the Lagrangian is maximised)
- Useful for understanding why a specific risk got a specific price

**Factor Tab** (ratebook mode):
- Per-factor before/after factor values
- Which factors changed most
- Heatmap of factor adjustments across levels

**Constraint Tab** (shared):
- Which constraints are binding vs slack
- Shadow prices (lambdas) — how much objective is being sacrificed per unit of constraint
- Sensitivity: how does the optimal solution change if a constraint is relaxed by 1%

### Frontier Interaction Flow

1. User configures objective, constraints, multiplier range
2. Clicks "Optimise" → batch solve runs, frontier generated
3. Frontier chart appears. User drags along the curve.
4. Each position updates the impact tab in real-time (re-solve at that threshold)
5. User picks a point → "Deploy" saves the lambdas/artifacts
6. Optimiser Apply node in the live pipeline uses the saved artifacts

## Library API (Draft)

```python
import numpy as np
from optim_library import OnlineOptimiser, RatebookOptimiser, SplineFitter

# --- Online Optimisation ---

# Fit splines (batched)
fitter = SplineFitter(knots=np.linspace(0.8, 1.2, 50))
objective_splines = fitter.fit(objective_values)       # (N, M) → spline coefficients
constraint_splines = fitter.fit(constraint_values)     # (N, M) → spline coefficients

# Solve
optimiser = OnlineOptimiser(
    multiplier_grid=np.linspace(0.8, 1.2, 50),
    chunk_size=500_000,
    max_iter=50,
    dtype=np.float32,
)

result = optimiser.solve(
    objective=objective_values,              # (N, M) float32
    constraints=[volume_values],             # list of (N, M) float32
    thresholds=[0.9],                        # volume ≥ 90%
)

result.lambdas          # (K,) — Lagrange multipliers
result.optimal_steps    # (N,) — index into multiplier grid per risk
result.optimal_values   # (N,) — optimal multiplier per risk
result.total_objective  # scalar — portfolio objective at optimum
result.total_constraints  # (K,) — portfolio constraint values at optimum

# Frontier
frontier = optimiser.frontier(
    objective=objective_values,
    constraints=[volume_values],
    threshold_range=(0.85, 0.98),
    n_points=20,
)

frontier.thresholds     # (n_points, K)
frontier.objectives     # (n_points,)
frontier.constraint_values  # (n_points, K)
frontier.lambdas        # (n_points, K)

# Live scoring
score = optimiser.apply(
    objective_curve=risk_objective_spline,   # single risk's spline
    constraint_curves=[risk_volume_spline],
    lambdas=result.lambdas,
)

score.optimal_multiplier  # scalar
score.optimal_objective   # scalar

# --- Ratebook Optimisation ---

ratebook_optimiser = RatebookOptimiser(
    candidate_range=(0.8, 1.2),
    candidate_steps=50,
    max_cycles=10,
    chunk_size=500_000,
    dtype=np.float32,
)

ratebook_result = ratebook_optimiser.solve(
    portfolio_residuals=residual_prices,     # (N,) — price without the factor being optimised
    factor_indices=factor_level_indices,     # (N, F) — which level each risk is at
    factor_values=current_factor_values,     # list of (L_f,) arrays
    objective_fn=objective_function,         # callable: (N,) prices → scalar
    constraint_fns=[volume_function],        # list of callables
    thresholds=[0.9],
)

ratebook_result.factor_values   # list of (L_f,) — optimised factor values
ratebook_result.lambdas         # (K,)
ratebook_result.total_objective # scalar
```

## Key Design Decisions

1. **Multiplier grid, not price grid**: uniform across all risks, simplifies spline fitting and memory layout.
2. **float32 throughout**: sufficient precision, halves memory.
3. **Chunked processing**: constant peak memory regardless of portfolio size.
4. **Lagrangian dual for both modes**: online uses it directly, ratebook uses it as the inner loop of coordinate descent. One core algorithm, two interfaces.
5. **Splines as the interchange format**: upstream models produce discrete points, the library fits splines, the splines are the artifact for live scoring.
6. **Library has no haute dependency**: pure numpy/scipy, independently usable and testable.

## Open Design Questions

1. **Live scoring for new risks**: a new quote wasn't in the batch. Options: (a) evaluate upstream models at M points on the fly and fit splines, (b) use segment-level average splines, (c) just use the lambdas with the multiplier grid evaluated at scoring time. Need to decide the default.

2. **Spline degree**: cubic splines are the default. Should we support linear interpolation as a faster/simpler option?

3. **Lambda convergence method**: subgradient (simple, may oscillate) vs bisection (tighter, more portfolio sweeps). Start with subgradient, add bisection option?

4. **Multiple objectives**: the current design is single-objective with constraints. Should we support multi-objective (Pareto across two objectives)? This changes the frontier from a curve to a surface.

5. **Constraint types**: currently only portfolio-level sum constraints (Σ ≥ threshold). Should we support per-risk constraints (e.g. max rate change per individual risk)? These are trivially handled by clipping the multiplier range per risk.

6. **Ratebook factor interactions**: coordinate descent optimises one factor at a time. Should we support optimising pairs of factors jointly (handles interactions but combinatorial cost grows)?
